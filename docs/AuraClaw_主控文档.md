# AuraClaw 主控文档

更新时间：2026-03-26  
状态：当前唯一开发依据

## 1. 这份文档的作用

这份文档不是讨论稿，也不是灵感记录。

它的作用只有一个：

> 作为 AuraClaw 当前唯一有效的产品主线和开发依据。

从现在开始：

- 页面结构按这份文档推进
- 内容模板按这份文档推进
- 新想法先补到这份文档，再决定是否写进代码
- 其他旧文档只保留为历史回溯或参考，不再并列作为“当前主线”

## 2. 一句话定位

> AuraClaw 帮普通人把一个刚装好的 OpenClaw，更快培养成真正会做事的助手，不是靠一堆零散 Prompt，而是靠一整段已经跑通过的正确经验。

再短一点可以说：

> AuraClaw 是 OpenClaw 的经验进化平台。

## 3. 我们到底在卖什么

AuraClaw 不卖下面这些东西：

- 不是教程百科
- 不是 Prompt 市场
- 不是 skill 导航站
- 不是“今天 AI 又能做什么”的资讯站

AuraClaw 真正卖的是：

> 已经被验证过的正确经验。

这些经验不是一句话，而是一整段可复制、可执行、可验证、可回退的说明。

对用户真正的交付层级应该是：

> 经验包 > skill > prompt

也就是说：

- `prompt` 只是最表层
- `skill` 只是经验包内部可能会用到的一个步骤
- `经验包` 才是用户真正需要拿走的东西

## 4. 核心世界观

AuraClaw 的底色不是“工具使用教程”，而是：

> 人类在使用 OpenClaw 的过程中积累正确经验，AuraClaw 把这些经验整理出来，再让更多人和更多 OpenClaw 一起继承它。

你把 OpenClaw 看作一种“硅基生命”，这个视角应该继续保留，因为它会直接决定 AuraClaw 的气质：

- 它不是冷冰冰的软件说明书
- 它是可以被培养、被增强、被逐步养成的存在
- AuraClaw 不只是网站，更像训练台、经验库和进化中介层

如果要把最近几轮构思收成一句产品语言，我建议固定成：

> AuraClaw 做的不是错误经验的堆积，而是正确做事经验的集体进化。

## 5. 用户真正遇到的问题

普通用户使用 OpenClaw 时，最常见的问题不是“不肯学”，而是：

- 不知道还存在某个关键能力
- 不知道要装什么
- 不知道别人已经验证过什么
- 不知道应该怎样把一句话说成真正可执行的任务
- 不知道任务失败时该怎么判断是能力问题还是表达问题

最典型的例子就是 `Scrapling`：

- 用户让 OpenClaw 去抓网页，可能像无头苍蝇一样反复尝试
- 但一旦知道有 Scrapling 这种抓取增强能力，体验就会直接跃迁

这说明很多问题不是不会问，而是根本不知道“还有这条能力路径存在”。

## 6. AuraClaw 的核心价值

AuraClaw 要同时完成 3 件事：

### 6.1 让用户更快拿到第一个结果

第一件事先做成，比第一天就理解整个生态更重要。

### 6.2 让 OpenClaw 真正变强

不是只教一句更好的话，而是告诉用户：

- 什么时候该装能力
- 该读哪个仓库
- 去哪下载
- 怎么验证
- 没成功时怎么退回

### 6.3 让用户自己的思维也变强

AuraClaw 的长期优势，不是替用户永远写提示词，而是：

- 训练用户的任务表达方式
- 训练用户的任务拆解能力
- 训练用户对“结果 / 约束 / 校验 / 回报 / 失败处理”的理解

也就是说，我们最终不只是“让 OpenClaw 变强”，还在“用 AI 的知识反过来训练人类”。

## 7. 当前内容结构

AuraClaw 当前按 4 个板块组织：

### 7.1 产后护理

目标：
- 帮新手先拿到第一个结果

特点：
- 低风险
- 高结果感
- 容易成功
- 中文场景友好

典型内容：
- 飞书会议纪要整理
- 日报 / 周报整理
- 学习资料转知识卡片

### 7.2 能力扩展

目标：
- 告诉用户 OpenClaw 还可以被增强

特点：
- 不是换一句话
- 是补一层能力
- 要把来源、下载、验证、回退讲清楚

典型内容：
- Scrapling 抓取增强
- GitHub 能力接入说明
- 未来可扩展到更多仓库 / skill / 工具链

### 7.3 对话训练

目标：
- 帮用户把模糊说法收成 OpenClaw 真能执行的结构

特点：
- 不是单纯“优化提示词”
- 是训练用户的大脑
- 要让用户慢慢学会自己说清楚

典型内容：
- 定时任务表达校准
- 网页采集表达校准

### 7.4 一人公司（OPC）

目标：
- 把单点能力接成完整工作骨架

特点：
- 更偏进阶
- 更偏场景化
- 更有传播性

典型内容：
- 一人内容公司骨架
- 一人情报公司骨架

## 8. AuraClaw 的最小内容单位：经验包

以后 AuraClaw 的内容单位不再叫“Prompt”或“配方页”，统一叫：

> 经验包

一个合格的经验包，至少要包含下面这些部分：

### 给用户看的部分

- 这包能帮我做成什么
- 我先要准备什么
- 适合谁先用
- 第一次建议怎么测试
- 成功长什么样

### 给 OpenClaw 执行的部分

- 需要用到什么能力 / skill / 仓库
- 官方来源是什么
- 下载或安装怎么做
- 验证流程是什么
- 正式任务开始前要先完成什么
- 如果失败怎么回退
- 装好后用户最短怎么说

### 给双方都重要的部分

- 风险等级
- 验收标准
- 下一步建议

经验包的原则是：

> 用户复制过去后，OpenClaw 应该能尽量复现这段经验，而不是只得到一句空泛指令。

## 9. 复制给 OpenClaw的内容，应该长什么样

以后页面里的“复制给 OpenClaw”不能再是这种东西：

`你现在是我的行业情报整理助手`

这种写法太像普通大模型 Prompt，不像 AuraClaw。

AuraClaw 给出去的，应该更像：

- 交给 OpenClaw 的执行说明
- 交给 OpenClaw 的能力接入说明
- 交给 OpenClaw 的训练说明

尤其是能力扩展类内容，至少要覆盖：

- 先不要乱试，先检查环境
- 先读哪个官方来源
- 什么时候安装
- 装完后先怎么验证
- 验证通过后再进入正式任务
- 如果失败，停在什么地方并如何回报

## 10. Human First, AI Ready

AuraClaw 首先是给人类用户用的。

所以首页第一屏必须先满足：

- 小白一眼能知道自己要做什么
- 一眼能知道怎么开始
- 文字尽量少，但路径要清楚

同时，AuraClaw 也要逐步做到 AI 可用。

这意味着：

- 人类入口应该尽量轻
- AI 入口应该尽量明确
- AI 不需要读全站，只需要读 `skill.md`

当前原则：

- `我是人类`：继续进入网站
- `我是 AI`：复制 `skill.md` 链接给 OpenClaw

## 11. 页面结构原则

AuraClaw 的页面不应该像说明文档，也不应该像 PPT。

页面原则固定为：

- 首屏少说话，只说最关键的判断
- 先引导用户做选择，再解释更多
- 一次只推动一个动作
- 不要把设计理念一股脑摊在首屏上
- 看起来像成品样板间，而不是头脑风暴板

首页当前应该承担的任务只有 3 个：

1. 说明 AuraClaw 是什么
2. 让用户知道怎么开始
3. 把用户导向最合适的经验包或路径

## 12. 当前 MVP 范围

当前 MVP 不追求“大而全”，只追求“第一眼顺、第一条能跑通”。

优先做深的，是最能体现 AuraClaw 差异化的经验包：

- 飞书会议纪要整理
- Scrapling 抓取增强
- 定时任务表达校准

下一批值得继续做深的内容：

- 抖音视频分析
- 行业情报简报
- GitHub 能力接桥
- 一人情报公司骨架

## 13. 内容来源规则

AuraClaw 后续可以参考、摘录、引用其他 OpenClaw 相关网站和社区内容，但原则固定：

- 先定模板，再填内容
- 不做整页搬运
- 以结构化重写为主
- 每条内容都要保留来源和原始链接

内容来源规则详见：

- `docs/reference/AuraClaw_内容来源模板与引用规则.md`

## 14. 当前不做什么

当前明确不做：

- 不做 AI 资讯站
- 不做泛泛的教程百科
- 不做只会堆 Prompt 的工具页
- 不做没有验证的内容堆砌
- 不做让用户必须先理解 skill 才能用的复杂系统

## 15. 开发规则

从现在开始，AuraClaw 按下面规则推进：

### 唯一主线

- 本文档是唯一主控文档
- 旧文档只用于历史回溯

### 先文档，后代码

- 新想法先补本文档
- 再改页面和内容

### 单条版本链

- 不另起炉灶
- 所有变更都沿现有主线迭代

### 体验优先级

- 先把 1 个经验包做深
- 再扩更多经验包
- 不要先堆很多浅内容

## 16. 当前状态判断

截至 2026-03-26，AuraClaw 当前处在：

> 世界观已经清楚，产品结构已经清楚，真正的壁垒也已经讲出来了，接下来最重要的是把“经验包”这条主线继续做深做稳。

一句话结尾：

> AuraClaw 不只是帮用户完成任务，而是在帮助用户训练 OpenClaw、训练自己，并把真实经验变成可以被更多人和更多 OpenClaw 继承的能力。
